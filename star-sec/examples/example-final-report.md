## 1. Executive summary

Acme App is a multi-tenant SaaS (Next.js frontend, NestJS API, PostgreSQL via Prisma, Stripe billing, deployed on a
container platform). Over a code-focused review with limited dynamic testing on staging, the application showed a
generally sound foundation — parameterized queries, framework-managed sessions, and secrets sourced from the platform
store — but **authorization is enforced inconsistently**, which produced the most serious findings. The headline risks
are a cross-tenant data exposure on the billing API and a payment webhook that does not verify signatures.

We recorded **8 findings**: 1 Critical, 2 High, 3 Medium, 1 Low, 1 Informational. The Critical and Highs concentrate on
the payments and billing paths and are realistically exploitable by an authenticated user. None require advanced skill

## 4. Security posture

Posture: **Needs Attention** — strong primitives undermined by inconsistent authorization. The team correctly uses an
ORM (no SQL injection observed), framework session management, and a secret manager. The recurring weakness is that
**object- and function-level authorization is applied ad hoc per handler** rather than centrally, so some endpoints
omit it. The payment integration trusts client-supplied state and lacks webhook signature verification. Fixing the authorization
pattern centrally would resolve multiple findings at once.

## 5. Attack surface

- **Public, unauthenticated:** marketing pages, `/api/auth/*` (login, signup, password reset), `/api/webhooks/stripe`.
- **Authenticated (any tenant user):** `/api/invoices/:id`, `/api/projects/*`, `/api/files/*`, realtime `/ws`.
- **Privileged:** `/api/admin/*` (tenant admins), internal `/metrics`.
- **Sensitive flows:** login/session, Stripe checkout + webhook fulfillment, file upload/download, tenant resolution.
- **Trust boundaries:** browser↔API, API↔Postgres, API↔Stripe, API↔Redis, API↔worker queue.

## 6. Findings by severity

### Critical

#### [PAY-001] Stripe webhook does not verify event signatures (forged payment events accepted)
**Severity:** Critical · **Priority:** P0 · **Confidence:** Confirmed · **Status:** Open
**Category:** Insufficient Verification of Data Authenticity — CWE-345 / A08:2021

The `/api/webhooks/stripe` handler parses the JSON body and fulfills orders based on `event.type` without calling the
provider's signature-verification step. An attacker who can reach the endpoint can POST a forged
`checkout.session.completed` event to mark an order paid and trigger fulfillment without paying.

```text
File: src/billing/webhook.controller.ts:31  (POST /api/webhooks/stripe)

  const event = req.body;                    // <-- raw JSON trusted directly
  if (event.type === 'checkout.session.completed') {
    await this.orders.fulfill(event.data.object.id);   // fulfillment on unverified event
  }
```

**Impact:** financial loss via free fulfillment; order/inventory integrity loss.
**Fix:** verify the signature against the raw request body using the webhook signing secret; reject on failure; add
idempotency keyed on the event ID; reconcile critical state against the provider API. **Test:** an unsigned/invalid
event returns `400` and does not fulfill.

### High

#### [API-007] Any authenticated user can read other users' invoices via `GET /api/invoices/:id`
**Severity:** High · **Priority:** P1 · **Confidence:** Confirmed · **Status:** Open
**Category:** Broken Object Level Authorization (IDOR/BOLA) — CWE-639 / API1:2023

Invoice lookup loads by primary key from the URL with no ownership/tenant scoping; sequential IDs make enumeration
trivial. (Full write-up in `example-finding.md`.) **Fix:** scope the query to the caller and return `404`; centralize
the check; audit sibling `/:id` endpoints.

#### [WEB-003] Open redirect on login `returnUrl` enables credential-phishing handoff
**Severity:** High · **Priority:** P1 · **Confidence:** High · **Status:** Needs Validation
**Category:** Open Redirect — CWE-601 / A01:2021

After login, the app redirects to a client-supplied `returnUrl` without validating it against an allowlist, so a
crafted link can bounce an authenticated user to an attacker-controlled site (useful in phishing chains).
**Fix:** allowlist relative paths / known hosts; reject absolute external URLs.

### Medium

- **[WEB-004] Missing security headers / weak CSP** (CWE-693) — no HSTS, no CSP; XSS impact would be amplified.
  *P2.* Add `helmet`-style headers and a strict CSP.
- **[API-009] No rate limiting on login and password-reset** (CWE-307/770) — enables slow brute force and reset abuse.
  *P2.* Add per-IP/per-account rate limits and lockout/backoff.
- **[INFRA-002] API container runs as root** (CWE-250) — increases blast radius of any RCE.
  *P2.* Add a non-root `USER`, drop capabilities, read-only FS.

### Low

- **[API-011] Verbose error responses leak stack traces in some 500s** (CWE-209) — minor info disclosure.
  *P3.* Return generic errors to clients; log details server-side.

### Informational

- **[OBS-001] No audit logging for admin actions** — recommend structured audit logs + alerting on authz failures.
  *P3.*

## 8. Suspected risks (need validation)

## 10. Secrets & configuration review
No live secrets committed. `.env.example` contains placeholders only. Configuration gaps: `DEBUG`-style verbose errors
in one path (API-011), missing security headers (WEB-004), and a root container (INFRA-002). Recommend a secret-rotation
runbook even though no exposure was found.

## 11. Architecture risks
Authorization is decentralized: each handler implements its own checks, so omissions (API-007) are easy to introduce.
The frontend trusts some values the backend should re-validate. Cross-repo seam: the webhook producer (Stripe) →
consumer (API) trust boundary is unverified (PAY-001). Recommended structural change: a single authorization policy
layer and idempotent, signature-verified webhook handling shared across billing flows.
